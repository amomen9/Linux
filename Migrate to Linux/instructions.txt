===============================================================================
 MIGRATE TO LINUX — REPRODUCIBILITY INSTRUCTIONS
===============================================================================

PURPOSE
-------
This file is a self-contained specification. An AI engine (or a human) should be
able to read ONLY this file — and regenerate, with the latest up-to-date data:

  1. detect_installed_windows_software.ps1   (Windows inventory + Linux rating)
  2. installed_windows_software.csv          (the report the script produces)
  3. <Target OS>/install_must_have_software.sh   (one unattended installer per
                                                  target-OS subdirectory)
  4. windows_settings_extract.ps1            (Windows settings scraper:
                                                  power/lid, display, keyboard,
                                                  telemetry/location, auto-update)
  5. <Target OS>/settings_config.sh          (per-distro apply functions +
                                                  dispatch table)
  6. <Target OS>/apply_settings.sh           (main runner: reads CSV,
                                                  applies settings via config)

Everything below reflects exactly what these artifacts must do. When data such as
"the best Linux alternative", version numbers, repository URLs or package names is
required, RESEARCH IT FRESH FROM THE WEB at generation time (the values baked into
the current files were researched in June 2026 and will drift).


DIRECTORY LAYOUT
----------------
  Migrate to Linux/
    instructions.txt                      <- this file (the spec)
    README.md                             <- human-facing overview
    windows_settings_extract.ps1          <- settings scraper (Windows)
    windows_configs.csv                   <- settings CSV (produced by PS1)
    detect_installed_windows_software.ps1 <- the inventory script (Windows)
    installed_windows_software.csv        <- generated report
    <Target OS>/                          <- one folder per intended target OS
        install_must_have_software.sh     <- generated software installer
        settings_config.sh                <- per-distro settings apply functions
        apply_settings.sh                 <- settings migration runner

  Each subdirectory is named after the intended target host OS, e.g.
  "Linux Mint (Ubuntu)". Add a new target by creating a new subdirectory and
  generating an installer in it per the INSTALLER SPEC below.


WORKFLOW (IMPORTANT — the human runs the PowerShell step)
---------------------------------------------------------
Step 1  The AI generates / updates detect_installed_windows_software.ps1.
Step 2  The HUMAN runs that .ps1 on the Windows machine being migrated:
            powershell -ExecutionPolicy Bypass -File detect_installed_windows_software.ps1
        (optionally with -Online). This produces installed_windows_software.csv.
        The AI must NOT fabricate the machine-derived rows; they come from this run.
Step 3  The HUMAN tells the AI to proceed with the CSV.
Step 4  The AI reads installed_windows_software.csv and generates / updates the
        per-target-OS installer (<Target OS>/install_must_have_software.sh) from the
        rows whose "Must be included on Linux" = yes, following the INSTALLER SPEC.
Step 5  The HUMAN runs the installer as root on the target OS.


===============================================================================
 PART A — POWERSHELL INVENTORY SCRIPT SPEC
 (file: detect_installed_windows_software.ps1)
===============================================================================

GOAL
  Produce a clean, de-duplicated inventory of all REAL installed Windows software
  and rate each app for migration to Linux. Output installed_windows_software.csv
  beside the script (UTF-8, -NoTypeInformation).

PARAMETERS
  -OutputPath <path>            default: installed_windows_software.csv beside script
  -MustIncludeThreshold <int>   default: 70   (see "Must be included" below)
  -IncludeSystemComponents      switch, default off
  -IncludeStoreApps <bool>      default: $true
  -Online                       switch, default off (live Repology lookups)

ERROR HANDLING (critical — must not crash on missing properties)
  Some registry keys returned by Get-ItemProperty and some Appx packages returned by
  Get-AppxPackage can lack properties that the script expects (e.g. a registry key
  with no values has only PS* provider properties, no DisplayName). Accessing a
  non-existent property can throw PropertyNotFoundException under certain PowerShell
  configurations. Therefore:
    * Wrap Get-ItemProperty and Get-AppxPackage calls in try/catch that returns @()
      on failure.
    * Every property access inside Where-Object and Select-Object blocks MUST be
      guarded with try/catch:
        try { $_.DisplayName } catch { $null = $dn }
        try { $_.IsFramework } catch { $false }
        try { $_.SignatureKind } catch { '' }
        ...and so on for every property accessed on objects from these cmdlets.
    * This applies to: DisplayName, SystemComponent, ParentKeyName, ParentDisplayName,
      WindowsInstaller, UninstallString, DisplayVersion, Publisher (Win32);
      IsFramework, SignatureKind, NonRemovable, Name, Version, Publisher (Appx).

COLLECTION (machine-derived; read live, never hand-authored)
  Win32 apps: enumerate these registry uninstall hives —
      HKLM:\Software\Microsoft\Windows\CurrentVersion\Uninstall\*
      HKLM:\Software\Wow6432Node\Microsoft\Windows\CurrentVersion\Uninstall\*
      HKCU:\Software\Microsoft\Windows\CurrentVersion\Uninstall\*   (do not omit HKCU)
    Keep rows with a non-empty DisplayName, and drop rows where SystemComponent=1,
    or ParentKeyName/ParentDisplayName is set, or it is an orphaned MSI row.
    Capture: Name=DisplayName, Version=DisplayVersion, Publisher, Source='Win32'.
  Store/UWP apps (when -IncludeStoreApps): Get-AppxPackage, dropping IsFramework,
    SignatureKind='System', NonRemovable, and a noise regex covering runtimes,
    media/image/speech extensions, vendor control panels, and OS plumbing.
    Make names human-friendly (e.g. Microsoft.WindowsCalculator -> "Windows
    Calculator") via a small override map plus a CamelCase splitter.

NOISE REMOVAL (the user's "no redundant entries" rule)
  Exclude Windows updates/hotfixes: KB numbers (also cross-check Get-HotFix),
    "Security Update", "Hotfix", "Service Pack", "Cumulative Update", "Update for".
  Unless -IncludeSystemComponents, also exclude redistributables, runtimes,
    drivers, and SDK/installer fragments (Visual C++ Redistributable, .NET runtimes,
    *Driver*, chipset/serial, ODBC/OLE DB drivers, WinFsp, OpenAL, VS Installer,
    Help Viewer, maintenance services, etc.). Keep these patterns in clearly-named,
    editable variables.

NAME CLEANING + DE-DUPLICATION
  Clean display names: strip "(remove only)", "(User)", trailing architecture
    parentheticals "(x64)/(64-bit)/(64 bit)" and trailing bare arch words.
  De-dup KEY keeps version digits (so genuinely distinct versions — e.g. three
    Python installs — remain separate) but drops vendor prefix, parentheticals and
    punctuation, so cosmetic Win32-vs-Store duplicates collapse. Prefer the Win32
    record when a collision occurs.

THE 9 CSV COLUMNS (exact header text, in this order)
  Name                        (machine)
  Version                     (machine)
  Publisher                   (machine)
  Source                      (machine: Win32 | Store)
  Linux Availability          (curated; flags — see below)
  Best Linux Alternative      (curated; see paid->free rule below)
  Alternative Competency      (curated; e.g. "100%", may exceed 100%)
  Pricing model               (curated; Free (FOSS) | Free | Freemium | Shareware | Paid)
  Must be included on Linux   (DERIVED in code; yes | no)

LINUX AVAILABILITY FLAGS — a cell may hold several, ALWAYS normalised to this order:
  Available on Linux ; Not Available ; Native Alternative ;
  Available as WebApp ; Linux Docker ; Windows Emulator (Wine/Proton)
  (Unknown apps get "Needs Review".)

ALTERNATIVE COMPETENCY
  Rough % estimate of how the recommended Linux app/alternative performs vs the
  Windows original. 100% = on par; <100% weaker; >100% the Linux option is better.
  MUST be filled for natively-available apps too (~100%, higher if Linux wins).

PRICING MODEL
  Pricing of the RECOMMENDED Linux option (native app or the named alternative).

PAID -> FREE RULE (do not omit)
  If a row's Pricing model is "Paid", the Best Linux Alternative cell must also name
  the best FREE alternative, appended right after the paid one, e.g.
    "...; free alternative: <free option>".

"MUST BE INCLUDED ON LINUX" (computed by the script, not hand-set)
  yes when ALL hold:
    * installable on Linux: Linux Availability contains "Available on Linux" OR
      "Native Alternative" (web-only / not-available => no, nothing to install);
    * Alternative Competency >= -MustIncludeThreshold (default 70);
    * it is the single most-competent build of that product — collapse duplicate
      installs of one product to ONE yes (identity key = name minus version, arch
      and generic suffix words; pick highest competency, Win32 preferred).

KNOWLEDGE BASE ($LinuxKB) — the single place curated data lives
  An ordered array; first matching pattern wins (specific before generic). Each row:
    @{ P='<regex matched against Name>';
       S='<availability flags>';
       A='<best alternative text>';
       C=<competency integer>;
       Pr='<pricing model>';
       F='<free alternative — only needed when Pr=Paid>' }
  To rate a new/unknown app: research the current best Linux option on the web and
  add an entry. Re-running the script regenerates the CSV; never hand-edit the
  machine columns in the CSV.

  IMPORTANT — PowerToys KB entry rules:
    PowerToys is NOT a single app on Linux. Its KB must list each feature's separate
    Linux replacement, with competency set to cover the overall replacement quality.
    The individual features and their Linux equivalents (researched June 2026):
      * Text Extractor (OCR)  -> Frog (tenderowl.com/frog, Flatpak: com.tenderowl.frog)
                                  shortcut-triggered OCR, select a screen region →
                                  extract text. This is the most important feature.
      * Keyboard accents      -> ibus-typing-booster (APT) or the Compose key (built-in)
      * Color Picker          -> gpick (APT) or KColorChooser — shortcut→click to pick hex
      * File Locksmith        -> fuser / lsof (APT) — CLI; GNOME File Locksmith for GUI
      * Find My Mouse         -> GNOME Settings > Mouse > "Show Pointer Location"
                                  (shake Ctrl to locate); KDE "Track Mouse" feature
      * Image Resizer         -> nautilus-image-converter (APT, right-click in Nautilus)
                                  or ImageMagick `convert` (CLI)
      * Mouse Without Borders -> Barrier / Input Leap (Flatpak: com.github.debauchee.barrier)
                                  — software KVM, share keyboard+mouse across PCs
      * ZoomIt                -> KMag (APT) or GNOME built-in zoom (Super+scroll)
      * Shortcut Guide        -> GNOME Super key overlay (hold Super) or KDE shortcuts
                                  cheat sheet
      * Run (launcher)        -> Ulauncher (APT, official PPA for Ubuntu 24.04)
    Set C=95 and Pr='Free (FOSS)' for PowerToys overall. The A field should concisely
    list each feature→replacement pair.

  IMPORTANT — Notepad++ KB entry rules:
    Notepad++ is classified as "Native Alternative; Windows Emulator (Wine/Proton)".
    The A field must list both native alternatives (notepadqq, geany, kate, VS Code)
    AND mention that the REAL Notepad++ runs well under Wine. The installer will
    install BOTH the native alternatives AND the real Notepad++ Windows installer
    under Wine. C=95.

OPTIONAL ONLINE MODE (-Online)
  For apps with no $LinuxKB match, query the Repology API
  (https://repology.org/api/v1/project/<slug>, custom User-Agent, be polite) to
  detect Linux packaging; cache results next to the CSV. This is the live
  "scrape the internet for latest data" path for unknowns.

OUTPUT + SUMMARY
  Export the 9 columns to CSV (UTF-8). Print totals: apps listed, excluded as
  noise, count of "yes", and a breakdown by availability category.


===============================================================================
 PART B — CSV CONSUMPTION
 (file: installed_windows_software.csv)
===============================================================================
  Treat the CSV as the source of truth for what to install on a target OS.
  The installer is built from rows where "Must be included on Linux" = yes,
  installing the native Linux app when "Linux Availability" includes "Available on
  Linux", otherwise the product named in "Best Linux Alternative" — noting that
  some entries (like PowerToys and Notepad++) name MULTIPLE alternatives that the
  installer must install individually.


===============================================================================
 PART C — PER-TARGET-OS INSTALLER SPEC
 (file: <Target OS>/install_must_have_software.sh)
===============================================================================

SCOPE
  One installer per target-OS subdirectory. It installs the Linux counterpart of
  EVERY CSV row flagged "Must be included on Linux" = yes. The example target is
  "Linux Mint (Ubuntu)" (Ubuntu/APT base + Flatpak). For a different target OS,
  translate the same intent to that OS's package managers and repositories.

EXECUTION MODEL
  * Bash, assumed to run as ROOT (verify EUID 0; exit otherwise).
  * Fully unattended (DEBIAN_FRONTEND=noninteractive on Debian/Ubuntu).
  * Idempotent: if an app is already installed, run its UPDATE path instead of
    reinstalling.
  * Never abort the whole run on one failure — wrap each install, record
    OK/FAIL/SKIP, and print a summary at the end.
  * CLEAN TERMINAL UI: save the real terminal on a spare fd (exec 3>&1 4>&2) and
    redirect everything else to a log file (exec >>"$LOG" 2>&1). Print ONLY the
    curated lines to fd 3 — "==> <step>" headers and indented "installed:/updated:"
    detail — so package-manager noise never reaches the screen. Prompts and the final
    summary also go to fd 3; show the log path at start and end.
  * LIVE PROGRESS (not just a spinner): while a step runs, show a SINGLE transient
    line on fd 3 that updates in place with real progress, and CLEAR it the moment that
    step's process terminates (leaving only its "installed:" result). For apt, get the
    actual percentage + current operation from APT::Status-Fd (run apt-get with
    -o APT::Status-Fd=9 redirected to a temp file and tail it). For non-apt commands
    (curl/conda/unzip/…), show the command's latest output line. This is essential for
    very long installs (e.g. texlive-full) so the user can see it is progressing.
  * TWO LOG FILES (both printed at the end):
      - install.log         : the full, verbose command output (for deep debugging).
      - install-results.log : a SHORT results log — one tab-separated line per step,
        "<timestamp> <status> <item> <reason>", where status is OK / FAIL / SKIP and
        <reason> is a concise failure reason. Capture the reason by teeing each step's
        output to a temp file inside the spinner helper and, on non-zero exit, pulling
        the last error-looking line (grep -iE 'error|unable|^E:|curl:|dpkg:' | tail).
        For a grouped package step, list the failed package(s) + their reason. End the
        results file with a totals line (OK / manual / skipped / failed counts).

AUTO-DETECT THE TARGET ENVIRONMENT (nothing hard-coded)
  Detect at runtime, on the machine the script is executing on, and use these
  everywhere a repo line, download URL or package name needs them — never hard-code
  amd64/x86_64 or a distro release:
    * Distro/base release codename + version: parse /etc/os-release; on Linux Mint use
      UBUNTU_CODENAME (fallback VERSION_CODENAME) and map it to the Ubuntu x.y number
      (e.g. jammy=22.04, noble=24.04) for codename/version-specific repos
      (Docker, PGDG, pgAdmin, Microsoft "prod").
    * Debian/apt architecture:  DEB_ARCH = `dpkg --print-architecture`
      (amd64 | arm64 | armhf | i386) — use in every apt "[arch=...]" source line and in
      vendor .deb URLs (Zoom zoom_${DEB_ARCH}.deb, GitKraken gitkraken-${DEB_ARCH}.deb,
      RStudio rstudio-latest-${DEB_ARCH}.deb, RealVNC .deb, etc.).
    * Kernel/tarball architecture:  KARCH = `uname -m` (x86_64 | aarch64 | ...), mapped
      as needed for non-.deb assets (e.g. Miniconda Miniconda3-latest-Linux-${KARCH}.sh).
      Use CONDA_ARCH for Miniconda, mapped from KARCH.
  If an app has no build for the detected architecture (some amd64-only vendor apps),
  let its download fail gracefully and be recorded as skipped (don't abort).

PPA / BACKPORTS FOR NEWER UBUNTU RELEASES
  On Ubuntu 24.04 (noble) specifically, some packages need extra PPAs because the
  vendor repos or main archives lag. Detect the codename and conditionally add:
    * dotnet/backports PPA — the Microsoft prod repo only carries .NET 8.0 on noble;
      .NET 9.0 SDK (dotnet-sdk-9.0) needs the ppa:dotnet/backports PPA added first.
    * Ulauncher PPA — ulauncher is not in the default noble repos; add
      ppa:agornostal/ulauncher via add-apt-repository.

TRUSTED SOURCES ONLY (hard rule)
  Every package, repository, key, .deb, script, and binary fetched by the installer
  MUST come from an absolutely trusted, verifiable source. Apply these checks to
  every source before including it in the generated .sh script:

    a) Official first-party domain — the source MUST be the software's OWN official
       domain (e.g. google.com for Chrome, microsoft.com for Edge/VS Code/PS/.NET,
       docker.com for Docker, postgresql.org for PostgreSQL/PGDG/pgAdmin, dbeaver.io
       for DBeaver, zoom.us for Zoom, discord.com for Discord, parsec.app for Parsec,
       gitkraken.com for GitKraken, rclone.org for rclone, anaconda.com for Miniconda,
       languagetool.org for LanguageTool, flathub.org for Flatpak, nodesource.com for
       Node.js, protonvpn.com for Proton VPN, anydesk.com for AnyDesk, winehq.org for
       WineHQ, virtualbox.org for Oracle VirtualBox, realvnc.com for RealVNC Viewer).

    b) HTTPS mandatory — all downloads, repos, and key URLs MUST use HTTPS. Reject
       any source that is HTTP-only or redirects to HTTP. (Exception: the distro's
       own APT repos which are GPG-signed and use trusted mirrors.)

    c) GPG signature verification — every third-party APT repo MUST add its keyring
       via `signed-by=/etc/apt/keyrings/<name>.gpg` in the sources.list line. Fetch
       keys over HTTPS with `curl -fsSL` and store in /etc/apt/keyrings/. The
       sources.list entry must use [signed-by=...] pointing to this keyring.

    d) No curl-pipe-bash from untrusted domains — `curl ... | bash` is acceptable
       ONLY from absolutely trusted first-party domains (e.g. rclone.org). Never pipe
       to bash from third-party sites, shortened URLs, or redirect services.

    e) Manual downloads (login-gated) — the script MUST print the EXACT official URL
       the user should visit, not a search suggestion or a third-party mirror. The
       URL must resolve to the vendor's own domain.

    f) SpotPlayer — this app is distributed from spotplayer.ir, a domain that cannot
       be verified as a trusted first-party source by the standards above. It MUST
       remain in the MANUAL section and the user MUST be warned that they are
       responsible for verifying the download's authenticity before installing it.

    g) Any source that cannot satisfy (a)–(c) MUST NOT be included in the automated
       section. Move it to MANUAL with a clear warning about verifying authenticity.
       Include a prominent "TRUSTED SOURCES GUARANTEE" comment block at the top of
       the generated script, after the header comment, stating that every source
       meets these criteria.

SOURCE-OF-PACKAGE PRIORITY (per the user's rule)
  1. The distro's NATIVE package repositories (APT on Mint/Ubuntu).
  2. The VENDOR's APT repo / .deb / official binary when the app is absent from the
     native repos OR the vendor ships a newer version.
  3. Flatpak (Flathub) only where neither of the above gives a clean result.
  Always install the LATEST available version.

DEPENDENCIES
  Install build/runtime deps that package managers won't pull automatically — in
  particular the kernel-module build chain for VMware:
    build-essential, gcc, make, dkms, linux-headers-$(uname -r), default-jdk (for
    Java-based tools such as LanguageTool), unzip, curl, wget, gnupg, gdebi, flatpak.
  Also install apt-transport-https, ca-certificates, lsb-release, software-properties-common.

POSTGRESQL (explicit requirement)
  Add the official PostgreSQL PGDG repository from postgresql.org
  (apt.postgresql.org, suite "<codename>-pgdg") and install the latest postgresql
  + postgresql-client.

WINE + NOTEPAD++ (explicit requirements)
  * Install WineHQ from the official winehq.org repository (add i386 architecture first).
  * After Wine is installed, additionally install the real Notepad++ Windows
    installer under Wine using the official GitHub release .exe (silent install
    with /S flag). Check for an existing Notepad++.exe in the Wine prefix first;
    skip re-installation if present (idempotent).
  * After Wine is configured, force Wine DPI to 192 (LogPixels = 0xC0) via
    `wine reg add` so fonts and UI are readable on high-DPI displays. Run this as
    the $SUDO_USER so it writes to their ~/.wine prefix.
  * The native Linux alternatives (notepadqq, geany) are also installed via APT.

POWERTOYS REPLACEMENTS (explicit requirements)
  PowerToys maps to MULTIPLE separate Linux packages, each covering one feature:
    * Text Extractor (OCR)     -> Frog          Flatpak: com.tenderowl.frog
    * Launcher (Run)           -> Ulauncher     APT (via PPA on noble)
    * Color Picker             -> gpick         APT
    * Image Resizer            -> nautilus-image-converter + imagemagick  APT
    * Mouse Without Borders    -> Barrier       Flatpak: com.github.debauchee.barrier

  Install EACH of these as individual steps, with descriptive labels like
  "PowerToys: OCR (Text Extractor)". For the info-only features, use `info` to
  print the configuration instructions.

    * Exec=xdg-open <URL> to open in the default browser
    * Use a relevant icon name from the system theme
    * Be owned by the SUDO_USER

REALVNC VIEWER (vendor .deb, not Flatpak)
  RealVNC Viewer is installed directly from the vendor's official .deb at
  realvnc.com (not on Flathub, not in APT). Use install_deb_url with the
  architecture-aware URL (downloads.realvnc.com, HTTPS).

PINTA (Flatpak on Ubuntu 24.04)
  Pinta was removed from the Ubuntu 24.04 (noble) APT repos. Install it via Flatpak
  (com.github.PintaProject.Pinta) instead of APT.

RSTUDIO (auto URL first, manual fallback)
  Try fetching RStudio from the direct vendor URL first (rstudio.org, HTTPS,
  architecture- and codename-aware). If that download fails (e.g. login-gated
  or URL changed), fall back to the manual prompt. Do NOT fail hard.

MANUAL / LOGIN-GATED DOWNLOADS (must come LAST)
  Some apps can't be fetched non-interactively (e.g. VMware Workstation needs a
  Broadcom login; SpotPlayer is a site download; Gurobi is license-gated; RStudio
  if its direct URL fails). For each, AFTER all automated installs:
    * print where to download it and the exact path to save it to;
    * loop, prompting the user, reading from /dev/tty, accepting "yes" (proceed only
      once the file exists) or "skip"; then install from the provided file.
  Put ALL such apps at the very end so the unattended part finishes first.

    support.broadcom.com, free account), SpotPlayer (.deb from spotplayer.ir with
    explicit warning about unverified source), Gurobi (commercial .tar.gz from
    gurobi.com, optional — free solvers glpk/cbc already installed), RStudio
    (fallback prompt if the direct URL failed).

  Detect the Ubuntu base codename even on Mint via UBUNTU_CODENAME in /etc/os-release
  and use it (plus DEB_ARCH/KARCH/CONDA_ARCH from "AUTO-DETECT THE TARGET ENVIRONMENT"
  above) for every codename- and architecture-specific repo line, download URL and
  package name.

FONT IMPORT (INTERACTIVE, AFTER POST-INSTALL, BEFORE MANUAL)

  After all automated installs complete and post-install services are configured,
  the script prompts the user to optionally import fonts from a local directory.
  This runs BEFORE the manual/login-gated downloads (section 11).

  Behaviour:
    * Print a banner explaining what the step does and which file types are
      supported.
    * Loop: ask the user for a directory path (or "skip") via /dev/tty.
    * Validate the path: must exist and be readable. If not, show a clear error
      and reprompt.
    * Recursively find all font files in the directory. Supported extensions
      (case-insensitive):
        .ttf .otf .ttc .woff .woff2 .pfa .pfb .afm .pfm .dfont .otb .bdf .pcf
        .gsf .otc .abf .chr .fnt .mxf
      These cover Windows TrueType/OpenType/Web fonts, macOS dfont, and Linux
      bitmap/pcf/bdf fonts.
    * If no matching files are found, warn the user and reprompt.
    * Copy each font file to /usr/local/share/fonts/user-import/ (create the
      directory first). Skip files that already exist with the same byte size
      (idempotent — safe to re-run).
    * If an individual file cannot be copied (permission denied, read error),
      warn about that file and continue with the rest. Track counts: total found,
      successfully installed, and failed.
    * After copying, run `fc-cache -fv` on the font directory to refresh the
      system font cache so applications (including Wine) see the new fonts
      immediately.
    * Log per-file status (installed / already present / copy-failed) to the
      install log via the script's `info`/`warn` functions.
    * Record the overall step as OK (with count summary), SKIP (user chose skip
      or no tty), or FAIL via the existing `mark_ok`/`mark_skip`/`mark_fail`
      mechanism.

  Error handling built in:
    * Directory doesn't exist          → print error, reprompt
    * Directory not readable           → print error, reprompt
    * No font files in directory       → warn + reprompt
    * Permission denied on a file      → skip that file, continue
    * No tty (non-interactive env)     → skip the entire step silently
    * Empty input or "skip"            → skip the entire step, record as SKIP

  The function name is `_install_fonts_from_dir` and it is invoked via:
    step "Import user fonts from directory" _install_fonts_from_dir

POST-INSTALL
  Enable services (ssh, postgresql, docker) and add the invoking $SUDO_USER to the
  docker group. Recommend a reboot at the end (kernel modules, group changes).


 PART D — SETTINGS MIGRATION SPEC
 (files: windows_settings_extract.ps1, <Target OS>/settings_config.sh,
         <Target OS>/apply_settings.sh)

GOAL
  Extract five categories of common Windows user settings, write them to
  windows_configs.csv (in the parent Migrate to Linux/ directory), then apply
  them on the target Linux distro using the per-distro scripts.

DIRECTORY LAYOUT (settings-migration files)
  Migrate to Linux/
    windows_settings_extract.ps1         <- PowerShell script run on Windows
    windows_configs.csv                  <- produced by the PS1, copied here
    <Target OS>/
        settings_config.sh               <- per-distro _apply_* functions +
                                            SETTINGS_DISPATCH table
        apply_settings.sh                <- main runner: reads CSV, dispatches

WORKFLOW (settings migration)
  Step S1  Run windows_settings_extract.ps1 on the Windows machine:
               powershell -ExecutionPolicy Bypass -File windows_settings_extract.ps1
           This produces windows_configs.csv in the same (Migrate to Linux/) folder.
  Step S2  Copy windows_configs.csv to the Migrate to Linux/ directory on the
           Linux machine.
  Step S3  On the target Linux machine, cd into the distro directory and run:
               sudo ./apply_settings.sh
           This sources settings_config.sh (same dir), reads ../windows_configs.csv,
           and calls the appropriate _apply_* function for each row.

CATEGORY 1 — POWER SETTINGS (lid-close action)

  Windows extract (windows_settings_extract.ps1):
    * Reads active power plan GUID from WMI (Win32_PowerPlan).
    * Queries Win32_PowerSettingDataIndex for the lid-close setting
      (GUID 5ca673a8-0e47-b2fd-5cb6-ef6e1e0a9c3e) on both AC and DC.
    * Registry fallback: HKLM\...\Power\User\PowerSchemes\...\ACSettingIndex / DCSettingIndex.
    * Produces CSV rows:
        Power | lid_close_on_ac      | <action>
        Power | lid_close_on_battery | <action>
      where <action> ∈ {do nothing, sleep, hibernate, shut down, turn off display}.

  Linux apply (settings_config.sh → _apply_power_lid):
    * Maps Windows action to logind action:
        do nothing → ignore, sleep → suspend, hibernate → hibernate,
        shut down → poweroff, turn off display → lock.
    * Writes HandleLidSwitch= and HandleLidSwitchExternalPower= in
      /etc/systemd/logind.conf.
    * Also writes a drop-in file /etc/systemd/logind.conf.d/50-migrate-lid.conf
      (survives distro upgrades).
    * Restarts systemd-logind.
    * Prints structured output: [OK] / [ERROR] / [INFO] per sub-step.

CATEGORY 2 — DISPLAY RESOLUTION & SCALING

  Windows extract:
    * Resolution from CIM_VideoController (CurrentHorizontalResolution ×
      CurrentVerticalResolution), e.g. "1920x1080".
    * Scaling from HKCU\Control Panel\Desktop\LogPixels or AppliedDPI,
      e.g. "125%" or "100%".
    * Produces CSV rows:
        Display | resolution | <WxH>
        Display | scaling    | <percentage>%

  Linux apply:
    * Resolution: xrandr --output <primary> --mode <WxH> if the mode exists.
    * Scaling: gsettings set org.gnome.desktop.interface text-scaling-factor and
      scaling-factor (integer only: ≥175%→2, ≥250%→3, ≥350%→4).

CATEGORY 3 — KEYBOARD LAYOUT & SHORTCUTS

  Windows extract:
    * Current culture from [System.Globalization.CultureInfo]::CurrentCulture.
    * Input methods from Get-WinUserLanguageList → LanguageTag list.
    * Notes standard Windows shortcuts (Win+E, Win+L, Win+Shift+S, etc.).
    * Produces CSV rows:
        Keyboard | layout         | <culture string>
        Keyboard | shortcuts_note | <semicolon-separated shortcut list>

  Linux apply:
    * Maps Windows language tags to XKB layouts (en-US→us, fa-IR→ir, etc.).
    * Applies via localectl set-x11-keymap (system-wide) and setxkbmap (session).
    * Shortcuts: prints informational reference (Win→GNOME equivalent table).

CATEGORY 4 — DISABLE TELEMETRY; LIMIT LOCATION

  Windows extract:
    * Telemetry level from HKLM\...\DataCollection\AllowTelemetry
      (0=Security, 1=Basic, 2=Enhanced, 3=Full).
    * Location consent from HKLM\...\CapabilityAccessManager\ConsentStore\location.
    * Produces CSV rows:
        Telemetry | telemetry_level  | <level>
        Telemetry | location_service | <enabled/disabled>

  Linux apply (telemetry):
    * Stops, disables, and masks services: whoopsie, apport, apt-daily.timer,
      apt-daily-upgrade.timer, motd-news.timer, packagekit.
    * Disables Ubuntu Pro ESM hook, apport crash reporting, MOTD news.
    * Purges packages: ubuntu-report, popularity-contest.

  Linux apply (location):
    * Stops, disables, and masks: geoclue.
    * gsettings set org.gnome.system.location enabled false.
    * Enables Wi-Fi MAC randomization in NetworkManager.

CATEGORY 5 — AUTO-UPDATE SERVICE FILES

  Windows extract:
    * Placeholder flag (no extractable Windows equivalent).
    * Produces CSV row:
        AutoUpdate | install_service_files | system_update

  Linux apply (_apply_auto_update_services):
    * Copies system_update.service and system_update.timer from
      <repo_root>/Scheduled systemd Automatic Update/Debian/service files/
      to /etc/systemd/system/.
    * systemctl daemon-reload.
    * systemctl enable --now system_update.timer.

SETTINGS_CONFIG.SH STRUCTURE
  The file is SOURCED by apply_settings.sh in the same distro directory. It must:
    * Define output helpers:
        _cfg_ok()    — green  [OK]    to stdout
        _cfg_info()  — dim    [INFO]  to stdout
        _cfg_error() — red    [ERROR] to stderr
    * Define one _apply_* function per config key (see dispatch table below).
      Each function takes the WindowsValue column as argument $1 and returns
      0 on success, non-zero on failure.
      Special functions: _apply_power_lid takes a second argument (logind key),
      _apply_keyboard_layout takes a second argument (input languages string).
    * Define the SETTINGS_DISPATCH associative array:
        ["Category|ConfigKey"] = "function_name"
    * Define the POWER_LOGIND_KEY associative array for lid-close key mapping.

APPLY_SETTINGS.SH STRUCTURE
  The main runner (same distro directory). It must:
    * Source settings_config.sh from $SCRIPT_DIR.
    * Read ../windows_configs.csv from the parent Migrate to Linux/ directory.
    * Parse the CSV header to get column indices (robust against reordering).
    * Iterate each data row, look up the dispatch key, call the function.
    * Track OK/FAIL/INFO and print a summary at the end.
    * Exit with code 1 if any step failed.
    * Must NOT use set -o pipefail (causes silent aborts).
    * Must use `rc=0; func || rc=$?` pattern, not set +e/set -e dance.

OUTPUT FORMAT (from _apply_* functions in settings_config.sh)
  Every _apply_* function must print structured output so the user sees exactly
  what succeeded and what failed. Example:

    [INFO]  Windows 'sleep' → logind HandleLidSwitch=suspend
    [OK]    removed old HandleLidSwitch= from logind.conf
    [OK]    wrote HandleLidSwitch=suspend → /etc/systemd/logind.conf
    [OK]    wrote /etc/systemd/logind.conf.d/50-migrate-lid.conf
    [OK]    restarted systemd-logind

    [INFO]  disabling Linux telemetry...
    [OK]    disabled & masked: whoopsie
    [INFO]  not present: apport
    [OK]    disabled ESM hook
    [OK]    purged: ubuntu-report

    [ERROR] xrandr not available or no monitors
